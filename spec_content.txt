— SECTION 00 —

Context &amp; Scope

WHAT THIS DOCUMENT COVERS

The Learning Centre Contextualisation Spec (March 2026) defined the full Stage 1 marketplace — 30 DEWA courses, 7 learning tracks, 10 reviews, and updated filters. This document defines Stage 2 (the learner&apos;s personal learning dashboard) and Stage 3 (the content governance dashboard for the EA Office) for the Learning Centre.

THE STAGE 3 GOVERNANCE MODEL

Stage 3 for the Learning Centre is not a fulfilment dashboard — it is a content governance dashboard. The EA Office does not directly edit course content. Instead, admins surface content items from the live platform, submit structured change requests (update a title, add a module, remove a course), and those changes are reviewed and approved before being applied. This ensures all content changes are governed and traceable.

WHAT THE EXISTING STAGE 2 AND STAGE 3 CONTAIN

The existing Stage 2 and Stage 3 for Learning Centre are generic versions built before the DEWA contextualisation. They do not reflect the 30 DEWA courses, 7 learning tracks, or division tags defined in the contextualisation spec. Both need to be updated to reflect the DEWA content.

— SECTION 02 —

Stage 3 — Content Governance Dashboard

The EA Office&apos;s content governance dashboard for the Learning Centre. Admin change request flow.

THE GOVERNANCE MODEL — HOW CONTENT CHANGES WORK

Stage 3 for the Learning Centre operates on a request-approve model. An admin (EA Office content manager) does not directly edit course content in the codebase or database. Instead, they use Stage 3 to surface existing content, identify what needs to change, and submit a structured change request describing the change they want made.

The change request is then reviewed and approved by a designated content approver (typically a senior EA Office member). On approval, the change request becomes an instruction for the developer (Hammton) to implement the change in the codebase. The platform surfaces the request and its approval status clearly.

WHY THIS MODEL

For DEWA&apos;s EA Office, this model ensures: (1) no accidental content changes — everything requires a second pair of eyes, (2) a full audit trail of what was changed, when, why, and who approved it, (3) the admin does not need technical access to the codebase — they describe the change in plain language and the developer implements it.

— SECTION 04 — CORE —

Stage 3 — Change Request Flow

How admins submit, approve, and implement content changes in the Learning Centre.

4.1 — Change Request Types

FIELD

SPECIFICATION

Update Course Metadata

Change the title, description, level, division tag, or duration of a course — without changing the course content itself

Update Course Content

Add a module, remove a module, update module content, reorder modules within a course

Add New Course

Request a new course to be added to the Learning Centre — with full course definition: title, level, description, modules, division tag

Unpublish Course

Remove a course from Stage 1 visibility — it remains in the Content Browser but learners can no longer see or enrol in it

Archive Course

Permanently remove a course — cannot be undone. Learners who were enrolled are notified.

Update Learning Track

Change a track&apos;s title, description, course sequence, duration, or certification details

Add New Learning Track

Request a new learning track — with full track definition: title, courses included, sequence, duration, certification details

Update Review

Edit or remove a learner review — requires justification

4.2 — Change Request Form

FIELD

SPECIFICATION

Subject

Pre-filled if raised from Content Browser — shows the course/track/review being changed. Manual entry if raised from the Change Requests page.

Change Type

Dropdown — all types from Section 4.1

Change Description

Textarea (required) — plain language description of exactly what should change. E.g. &apos;Change the course title from X to Y&apos; or &apos;Add a new module after Module 3: [module title and description]&apos; or &apos;Remove Module 5 — content is outdated&apos;.

Reason

Textarea (required) — why this change is needed. E.g. &apos;Course title does not reflect updated DEWA EA 4.0 terminology&apos; or &apos;Module 5 references a system that was decommissioned in Q4 2025&apos;.

Urgency

Dropdown — Low | Medium | High | Critical

Attachments

Optional file upload — admin can attach reference documents, updated content, images, or supporting materials for the developer to use when implementing

Preview of current content

Read-only display of the current content being changed — so the approver can compare current vs requested change side by side

4.3 — Change Request Status Flow

STATUS FLOW

FIELD

SPECIFICATION

Draft

Admin has started but not submitted the change request

Submitted

Change request submitted — awaiting approval. Appears in Pending Approval queue for Content Approvers.

Approved

Content Approver has approved the change. Request is now ready for developer implementation. Developer is notified.

Rejected

Content Approver has rejected the change — with mandatory rejection reason. Admin is notified. Admin can revise and resubmit.

In Implementation

Developer has acknowledged the change and is implementing it

Implemented

Developer has marked the change as implemented. Admin is notified to verify.

Verified

Admin confirms the change has been correctly implemented. Request is closed.

Closed — Not Verified

Admin raises a query — implementation does not match the request. Goes back to In Implementation.

4.4 — Pending Approval Page (Content Approver role)

FIELD

SPECIFICATION

Queue entry fields

Subject (course/track/review title) | Change type badge | Submitted by | Submitted date | Urgency badge | Open button

Open request detail

Shows full change request: current content preview, change description, reason, urgency, attachments

Approve action

Green &apos;Approve&apos; button — required approval note (optional but encouraged). Status: Submitted → Approved. Admin notified. Developer notified.

Reject action

Red &apos;Reject&apos; button — mandatory rejection reason. Status: Submitted → Rejected. Admin notified with reason.

Request Clarification

Amber button — Content Approver can ask the admin a question before deciding. Status: Submitted → Clarification Requested. Admin notified.

4.5 — Developer Implementation View

HOW THE DEVELOPER (HAMMTON) SEES AND ACTS ON APPROVED CHANGES

When a change request is approved, it becomes an implementation instruction for the developer. The platform surfaces approved changes in a clear, actionable format that can be used directly as a coding prompt.

FIELD

SPECIFICATION

Approved change view

Shows: Change Type, Subject, exact Change Description, Reason, Attachments, Approval note, Approved by, Approved date

Implementation prompt

A formatted, copy-ready instruction block that the developer can paste directly into Cursor/Kiro — pre-formatted as a technical instruction based on the change description

Mark as Implemented

Developer clicks &apos;Mark as Implemented&apos; once the change is live. Status: Approved → Implemented. Admin notified to verify.

Implementation notes

Developer can add implementation notes — e.g. &apos;Changed in CoursesData.ts line 47, course ID LC-DEWA-001&apos;

NOTE: For the prototype, the Implementation Prompt block does not need to auto-generate. It can be a styled display of the change description formatted as a developer instruction. The key functional requirement is the Mark as Implemented action and the status update it triggers.

— SECTION 06 —

End-to-End Flows

Two flows: (1) Learner journey Stage 1 to Stage 2. (2) Admin content change request flow Stage 3.

FLOW 1 — LEARNER JOURNEY

STEP 1 Browse Learning Centre [Learner]

Learner navigates to Learning Centre Stage 1. Browses courses by tab — Courses &amp; Curriculum, Learning Tracks, or Reviews.

▼

STEP 2 Find a course [Learner]

Learner filters by Division (e.g. Transmission) or Level (Beginner). Finds a relevant course. Clicks the card.

▼

STEP 3 Enrol in course [Learner]

Course detail page opens. Learner clicks &apos;Enrol&apos;. Course added to My Courses in Stage 2. Learner can start immediately.

▼

STEP 4 Track progress in Stage 2 [Learner]

Learner navigates to Stage 2 → My Courses. Sees course with 0% progress. Clicks &apos;Continue&apos; to return to course content.

▼

STEP 5 Complete course [Learner]

Learner works through modules. Progress updates after each module. On final module completion — course marked Complete. Certificate issued if applicable.

▼

STEP 6 View certificate [Learner]

Learner navigates to Stage 2 → Certificates. New certificate visible. Downloads PDF.

▼

STEP 7 Enrol in a Learning Track [Learner]

Learner navigates to Stage 1 → Learning Tracks. Finds DEWA EA Practitioner Pathway. Clicks &apos;Enrol in Track&apos;. All 7 courses added to My Courses.

▼

STEP 8 Track progression in My Tracks [Learner]

Stage 2 → My Tracks shows the track with X of 7 courses complete. Progress bar updates as individual courses are completed.

▼

FLOW 2 — ADMIN CONTENT CHANGE REQUEST

STEP 1 Admin accesses Stage 3 [Admin]

Admin navigates to Learning Centre Stage 1 and clicks the TO Contributor card. Enters Stage 3 Content Governance Dashboard.

▼

STEP 2 Admin browses Content Browser [Admin]

Admin opens Content Browser → Courses &amp; Curriculum. Sees all 30 DEWA courses with admin metadata. Finds a course that needs updating.

▼

STEP 3 Admin raises change request [Admin]

Admin clicks &apos;Raise Change Request&apos; on the course. Change request form opens — subject pre-filled with the course. Admin selects Change Type (e.g. &apos;Update Course Metadata&apos;), describes the change (&apos;Change the course title from X to Y&apos;), gives the reason, sets urgency. Submits.

▼

STEP 4 Request in Pending Approval [System]

Change request appears in Pending Approval queue. Content Approver is notified.

▼

STEP 5 Content Approver reviews [Approver]

Content Approver opens the request. Sees current content and the requested change side by side. Reads the reason. Clicks &apos;Approve&apos;.

▼

STEP 6 Developer notified [System]

Developer (Hammton) is notified of approved change. Change request shows in Approved Changes with the formatted implementation instruction.

▼

STEP 7 Developer implements [Developer]

Developer reads the change instruction, implements the change in the codebase (e.g. updates the course title in CoursesData.ts). Clicks &apos;Mark as Implemented&apos;. Adds implementation note.

▼

STEP 8 Admin verifies [Admin]

Admin is notified. Reviews the change in Stage 1 to confirm it matches the request. Clicks &apos;Verified&apos;. Request is closed.
