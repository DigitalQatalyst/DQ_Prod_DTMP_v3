**1\. DTMP PLATFORM CONTEXT** 

**What is DTMP?** 

The **Digital Transformation Metamodel Platform (DTMP)** is an internal enterprise platform designed to accelerate an organization's digital transformation journey toward becoming a Digital-Centric Organization (DCO). 

DTMP provides: 

- **9 specialized marketplaces** for transformation resources and services 
- **AI-powered intelligence** and analytics across the digital estate 
- **Governance frameworks** aligned to the 4D Model (Discern, Design, Deploy, Drive) 
- **Role-based access** for different stakeholder groups (Staff, Managers, Directors, Executives, Partners) 

**Platform Architecture** 

DTMP operates in two stages: 

**Stage 1: Marketplaces (Public/Pre-login)** 

- Browse and discover available services and resources 
- Explanatory content to help users understand what's available 
- Search and filter capabilities 
- Call-to-action to access detailed functionality 

**Stage 2: Service Hub (Authenticated/Post-login)** 

- Access detailed dashboards, tools, and workspaces 
- Execute on selected services 
- Collaborate with Transformation Office 
- Role-based access to data and capabilities 
- **Layout:** LVE (Left-Vertical-Expanded)  
- **Column 1:** Sidebar navigation (all marketplaces) 
- **Column 2:** Resource/content list or categories 
- **Column 3:** Detailed content viewer or workspace 

**The 9 Marketplaces** 

1.  **Learning Center** - Courses, certifications, training materials 
2.  **Knowledge Center** - Documentation, guides, best practices, case studies 
3.  **Portfolio Management** - Application inventory and catalog 
4.  **Support Services** ← **THIS MARKETPLACE** 
5.  **Digital Intelligence** - AI-powered analytics dashboards 
6.  **Solutions Specifications** - Blueprints, reference architectures, design patterns 
7.  **Solutions Build** - Request delivery of pre-built or custom solutions 
8.  **Lifecycle Management** - Active transformation initiatives (retire, modernize, replace) 
9.  **AI DocWriter/Templates** - AI-generated governance documents (optional) 

**Transformation Office (TO)** 

The Transformation Office is the central team that: 

- Governs the DTMP platform 
- Provides services through the marketplaces 
- Manages transformation initiatives 
- Maintains data quality and standards 
- **Supports users across all marketplaces through Support Services**

**2\. THIS MARKETPLACE: SUPPORT SERVICES** 

**What is Support Services?** 

Support Services is the **operational support hub** for all DTMP-related activities and transformation initiatives. It provides a centralized system for **service requests, incident management, problem tracking, and operational support** across the digital transformation journey. 

**Core Function:** Enable users to request help, report issues, track incidents, and receive support from the Transformation Office and specialized support teams for any transformation-related needs. 

**Why Does This Marketplace Exist?** 

**Problem Being Solved:** 

Organizations typically struggle with: 

- ❌ **Fragmented support channels** - Users don't know where to go for help (email, Slack, phone, walk-ups) 
- ❌ **Lost requests** - Support requests sent via email get buried, forgotten, or lost 
- ❌ **No visibility** - Users can't track status of their requests 
- ❌ **Duplicate work** - Multiple people request the same thing, wasting effort 
- ❌ **No prioritization** - Urgent issues buried among low-priority requests 
- ❌ **Knowledge loss** - Solutions to problems not captured for future reference 
- ❌ **No metrics** - Can't measure response times, resolution rates, satisfaction 
- ❌ **Poor handoffs** - Requests passed between teams without context 

**Solution:** 

Support Services provides: 

- ✅ **Single point of contact** - One place to request help for anything DTMP-related 
- ✅ **Structured ticketing** - All requests tracked in system with unique IDs 
- ✅ **Self-service portal** - Users track their own requests in real-time 
- ✅ **Categorized requests** - Service requests, incidents, problems, questions 
- ✅ **SLA management** - Priority-based response and resolution times 
- ✅ **Knowledge capture** - Solutions documented for future reference 
- ✅ **Assignment & routing** - Requests automatically routed to right team 
- ✅ **Metrics & reporting** - Measure performance, identify trends 

**Support Services vs. Other Marketplaces** 

**Key Distinction:** Support Services is **operational support**, not content or tools. 

|     |     |
| --- | --- |
| **What It Is** | **What It's NOT** |
| Request help with using DTMP | Learning how to use DTMP (Learning Center) |
| Report issues/bugs | Reading documentation (Knowledge Center) |
| Ask questions | Searching for answers (Knowledge Center) |
| Get assistance from TO | Automated intelligence (Digital Intelligence) |
| Request account access | Managing your own account settings |

**When to use Support Services:** 

- "I can't access the Portfolio Management tool" 
- "I need help interpreting this intelligence dashboard" 
- "This course video won't play" 
- "I need my manager to approve my stage gate submission" 
- "Request access to Solutions Specifications" 
- "Report a bug in the Knowledge Center search" 

**When NOT to use Support Services:** 

- "How do I design microservices?" → Learning Center 
- "What's the API naming convention?" → Knowledge Center 
- "Show me our application portfolio" → Portfolio Management

**What's Inside This Marketplace?** 

Support Services provides **support channels and request types**: 

**Request Types:** 

1.  **Incident** - Something is broken and needs immediate fixing  

- System is down 
- Error messages 
- Can't access a feature 
- Data is missing/incorrect 
- Performance issues 
- **Priority:** High to Critical 
- **SLA:** 4 hours response, 24 hours resolution (P1) 

1.  **Service Request** - Request for something to be done  

- Request access to a tool 
- Request data update 
- Request dashboard customization 
- Request new feature 
- Request documentation 
- **Priority:** Medium to Low 
- **SLA:** 1 business day response, 5 business days resolution 

1.  **Question** - Need information or clarification  

- How do I...? 
- Why is...? 
- When will...? 
- Who can help with...? 
- **Priority:** Low 
- **SLA:** 4 hours response, 1 business day resolution 

1.  **Problem** - Recurring issue or root cause investigation  

- Multiple incidents with same cause 
- Systemic issue requiring investigation 
- Pattern of failures 
- **Priority:** Medium 
- **SLA:** 2 business days response, 10 business days resolution 

1.  **Change Request** - Request to change/improve DTMP  

- Feature enhancement 
- Process improvement 
- New capability 
- **Priority:** Low to Medium 
- **SLA:** 3 business days response, evaluated for roadmap 

**Support Categories:** 

- **Portfolio Management** - Support for application portfolio tools 
- **Learning Center** - Course access, enrollment, certificates 
- **Knowledge Center** - Article access, content issues 
- **Digital Intelligence** - Dashboard access, data issues 
- **Solutions Specifications** - Blueprint access, spec issues 
- **Solutions Build** - Build request support, delivery issues 
- **Lifecycle Management** - Stage gate support, initiative tracking 
- **Platform/Account** - Login, access, permissions 
- **Other** - General questions or uncategorized 

**Priority Levels:** 

- **P1 - Critical:** System down, can't work, revenue impact 
- **P2 - High:** Major feature broken, workaround exists 
- **P3 - Medium:** Minor issue, annoying but not blocking 
- **P4 - Low:** Question, enhancement request, minor bug 

**Status Values:** 

- **New** - Just submitted, not yet reviewed 
- **Open** - Acknowledged, assigned to support agent 
- **In Progress** - Being actively worked on 
- **Pending** - Waiting for user response or external dependency 
- **Resolved** - Issue fixed, awaiting user confirmation 
- **Closed** - Confirmed resolved by user 
- **Cancelled** - Request withdrawn by user

**9\. TECHNICAL IMPLEMENTATION NOTES** 

**Ticket Numbering** 

**Format:** SUP-YYYY-NNNN 

- SUP = Support 
- YYYY = Year (4 digits) 
- NNNN = Sequential number (auto-increment) 

**Example:** SUP-2026-1247 

**Implementation:** 

- Database sequence/counter 
- Reset annually or continuous 
- Ensure uniqueness 

**SLA Calculation** 

**Business Hours:** 

- Monday-Friday, 8:00 AM - 6:00 PM EST 
- Exclude weekends and holidays 
- P1 uses 24/7 calendar (critical issues) 

**Algorithm:** 

typescript 

function calculateSLA( 

&nbsp; createdAt: Date, 

&nbsp; priority: string, 

&nbsp; businessHoursOnly: boolean 

): { responseTarget: Date; resolutionTarget: Date } { 

&nbsp; const sla = getSLAConfig(priority); 

&nbsp;  

&nbsp; const responseTarget = addBusinessHours( 

&nbsp;   createdAt, 

&nbsp;   sla.responseTime, 

&nbsp;   businessHoursOnly 

&nbsp; ); 

&nbsp;  

&nbsp; const resolutionTarget = addBusinessHours( 

&nbsp;   createdAt, 

&nbsp;   sla.resolutionTime, 

&nbsp;   businessHoursOnly 

&nbsp; ); 

&nbsp;  

&nbsp; return { responseTarget, resolutionTarget }; 

}

**US-002:** As a user, I want to attach screenshots to my request so the support agent can see the error. 

**US-003:** As a user, I want the system to suggest relevant KB articles as I type so I can self-serve if possible. 

**Epic 2: Request Tracking** 

**US-004:** As a user, I want to see all my support requests so I can track their status. 

**US-005:** As a user, I want to receive notifications when my ticket is updated so I stay informed. 

**US-006:** As a user, I want to add comments to my ticket so I can provide additional information. 

**Epic 3: Request Resolution** 

**US-007:** As a user, I want to mark my issue as resolved when the solution works so the ticket can be closed. 

**US-008:** As a user, I want to rate my support experience so the team knows how they're doing. 

**US-009:** As a user, I want to reopen a ticket if the issue comes back so I don't have to start over. 

**Epic 4: Support Agent Work** 

**US-010:** As a Support Agent, I want tickets automatically assigned to me based on my expertise so I can work efficiently. 

**US-011:** As a Support Agent, I want to add internal notes to tickets so I can document investigation without showing users. 

**US-012:** As a Support Agent, I want to see SLA deadlines so I can prioritize my work. 

**Epic 5: Manager Oversight** 

**US-013:** As a Manager, I want to see my team's support tickets so I know what issues they're facing. 

**US-014:** As a Manager, I want to see SLA compliance metrics so I can ensure we're meeting commitments. 

**US-015:** As a Manager, I want to identify common issues so I can address root causes.

**15\. APPENDIX** 

**Glossary** 

- **Incident:** Something broken that needs fixing 
- **Service Request:** Request for something to be done 
- **Problem:** Root cause investigation for recurring incidents 
- **Change Request:** Request to modify or improve DTMP 
- **SLA:** Service Level Agreement (commitment to response/resolution time) 
- **FCR:** First Contact Resolution (resolved on first interaction) 
- **Deflection:** User finds answer via self-service, doesn't submit ticket 
- **TO:** Transformation Office 
- **LVE:** Left-Vertical-Expanded (layout pattern) 

**Support Ticket Lifecycle** 

New → Open → In Progress → Resolved → Closed 

&nbsp;        ↓         ↓            ↓ 

&nbsp;     Pending   Pending      (Can reopen) 

&nbsp;                             ↓ 

&nbsp;                         Reopened → Open 

**Priority Guidelines** 

**P1 - Critical:** 

- System completely down 
- Can't work at all 
- Revenue impact 
- Security breach 

**P2 - High:** 

- Major feature broken 
- Workaround exists but difficult 
- Significant productivity impact 

**P3 - Medium:** 

- Minor issue 
- Annoying but not blocking 
- Low productivity impact 

**P4 - Low:** 

- Cosmetic issue 
- Question 
- Enhancement request 
- No urgency
